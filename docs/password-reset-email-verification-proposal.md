## 1. 배경 — 현재 구현의 문제

지금 `POST /auth/forgot-password` 엔드포인트는 이메일만 입력받아 즉시 임시 비밀번호를 응답으로 반환한다.

```
공격자 → POST /auth/forgot-password (email: victim@x.com)
       ← 200 OK { temp_password: "Xy@k3..." }
```

### 보안 취약점

| 문제 | 영향 |
|---|---|
| 이메일만 알면 누구나 임시 비밀번호 발급 가능 | **계정 탈취** — 공격자가 즉시 로그인 |
| 진짜 사용자는 기존 비번 무효화로 로그인 불가 | 서비스 거부 (DoS) |
| 응답으로 비밀번호 노출 → 로그/네트워크 캐시에 잔존 위험 | 2차 유출 |
| 가입 여부가 응답으로 노출됨 | 사용자 이메일 열거 공격 가능 |

### 현 상태가 임시 구현인 이유

MVP 단계에서 SMS·이메일 인프라가 없어 "기능 자체는 동작하도록" 넣어둔 데모용 구현. 운영 전 반드시 교체 필요.

## 2. 표준 흐름 (제안)

이메일 인증 코드 방식. 가장 일반적이고 OWASP 권장 흐름이다.

```
[1단계] 사용자가 이메일 입력 → "재설정 요청"
   └→ 서버: 6자리 코드 생성 → Redis에 5분 TTL로 저장
   └→ 서버: 사용자 이메일로 코드 발송
   └→ 응답: 항상 200 "전송했습니다" (가입 여부 노출 X)

[2단계] 사용자가 이메일에서 코드 확인 → 코드 입력
   └→ 서버: Redis에서 코드 조회·검증
   └→ 응답: 일회용 리셋 토큰 발급 (5분 유효)

[3단계] 사용자가 새 비밀번호 입력
   └→ 서버: 리셋 토큰 검증 → 비번 변경 + 코드/토큰 삭제
```

### 핵심 보안 장치

- **코드는 이메일을 받을 수 있는 사람만 본다** → 본인 확인 성립
- **응답은 항상 동일 ("전송했습니다")** → 가입 이메일 vs 미가입 이메일 구별 불가
- **TTL 5분** → 코드 누출돼도 금방 무효
- **시도 횟수 제한 (예: 5회)** → 코드 무차별 대입 차단
- **리셋 토큰은 일회용** → 재사용 차단

## 3. 필요한 인프라 — 이메일 발송 서비스 선택

> 모든 옵션이 **무료 한도 내** 사용 가능. 우리 프로젝트 규모(부트캠프 발표용)에선 차고 넘침.

### 비교표

| 서비스 | 무료 한도 | 신용카드 등록 | 발신자 주소 | 추천 상황 |
|---|---|---|---|---|
| **Resend** ⭐️ | 100통/일, 3,000통/월 | **불필요** | `onboarding@resend.dev` 또는 자체 도메인 | 깔끔하고 부담 적음 |
| **SendGrid** | 100통/일 (계속) | 가입 시 요구 (한도 내 청구 X) | 자체 도메인 인증 필요 | 업계 표준 |
| **Gmail SMTP** | 500통/일 | **불필요** | 본인 Gmail | 가장 쉬우나 발표 시 어색 |
| **Brevo (구 Sendinblue)** | 300통/일 | 불필요 | 자체 도메인 | |
| **AWS SES** | $0.10/1000통 | 필요 | 자체 도메인 | 배포 후 비용 최적화 |
| **Naver Cloud SENS** | 종량제 | 필요 | 한국 도메인 가능 | 국내 서비스 선호 시 |

### 풀스택 의견 (논의용 초안)

**Resend 추천.**

이유:
- 가입 즉시 사용 가능 (신용카드 X, 도메인 인증 X)
- 발신자가 `onboarding@resend.dev` 로 표시되어 발표 시연에 자연스러움
- API가 간결해서 구현 코드가 짧음 (5줄 수준)
- 3,000통/월 무료 → 졸업 프로젝트 전체 사용량 대비 차고 넘침

대안:
- **Gmail SMTP**: 가장 쉬우나 발신자가 개인 이메일(`rladbsrl298@gmail.com`)이라 발표 시연에 부적절
- **SendGrid**: 안정성·업계 표준이지만 가입 절차가 길고 신용카드 요구
- 도메인을 살 계획이 없다면 Resend가 가장 무난

## 4. 작업량 예상

### 백엔드 (3~5시간)

- 새 모델/저장소: `PasswordResetCode` (Redis 활용 권장)
- 새 서비스: `EmailService.send_reset_code(email, code)`
- 엔드포인트 재설계:
  - `POST /auth/forgot-password` — 코드 발송 (응답 항상 200)
  - `POST /auth/verify-reset-code` — 코드 검증 + 리셋 토큰 발급
  - `POST /auth/reset-password` — 리셋 토큰 + 새 비번으로 변경
- 기존 `issue_temp_password` 제거 (취약점이므로)
- 시도 횟수 제한 미들웨어 또는 Redis 카운터

### 프론트엔드 (1~2시간)

- 1단계 화면: 이메일 입력
- 2단계 화면: "이메일을 확인하세요" + 6자리 코드 입력
- 3단계 화면: 새 비밀번호 입력
- 단계 전환 상태 관리

### 환경변수 추가

```env
RESEND_API_KEY=re_xxxxxxxxxx
EMAIL_FROM_ADDRESS=onboarding@resend.dev
PASSWORD_RESET_CODE_TTL_SECONDS=300
PASSWORD_RESET_MAX_ATTEMPTS=5
```

### 의존성 추가

- 백엔드: `resend` Python SDK (또는 `httpx`로 REST 직접 호출)
