## 1. 한 줄 요약

> **백엔드에서 발생하는 모든 에러 응답을 자동으로 한국어로 변환하는 글로벌 미들웨어를 추가했습니다.**
> 앞으로 새 엔드포인트·DTO를 만들 때 영어 에러 메시지가 사용자에게 노출될 걱정이 없습니다.

---

## 2. 왜 만들었나

### 문제
지금까지는 로그인·회원가입 등에서 잘못된 값 입력 시 **빨간 에러가 영어로 떴어요**. 사용자(특히 고령자·비전공자)에게 친화적이지 않은 문제.

예시:
```
❌ Before: "value is not a valid email address"
❌ Before: "field required"
❌ Before: "Internal Server Error"
```

### 원인 분석
에러가 영어로 뜨는 곳은 **개발자가 한국어로 안 적어서가 아니라 시스템이 자동 생성**하기 때문이었어요.

| 출처 | 영어 이유 |
|---|---|
| Pydantic 422 검증 에러 | FastAPI 기본값 (Pydantic 라이브러리) |
| FastAPI 라우터 매칭 실패 | Starlette 기본값 |
| HTTP 표준 코드 | Starlette 기본값 |
| 예상치 못한 예외 (500) | Python·OS 메시지 |

### 일회성 픽스의 한계
"지금 보이는 에러만 한국어로 바꾸자"로 접근하면 **새 기능 추가 시 또 영어가 나옴**. 그래서 **인프라 레벨에서 자동 처리**하는 방식을 선택했습니다.

→ **"한 번 만들어 두면 영원히 자동 한국어"** 시스템을 만들었어요.

## 3. Before vs After 비교

### 🅐 회원가입에서 잘못된 이메일 입력 시

**Before:**
```json
{
  "detail": [
    {
      "loc": ["body", "email"],
      "msg": "value is not a valid email address",
      "type": "value_error.email"
    }
  ]
}
```

**After:**
```json
{
  "detail": [
    {
      "field": "이메일",
      "message": "올바른 이메일 형식이 아닙니다."
    }
  ]
}
```

### 🅑 필수 필드 누락

**Before:** `"field required"`
**After:** `"필수 입력 항목입니다."`

### 🅒 비밀번호가 너무 짧음

**Before:** `"ensure this value has at least 8 characters"`
**After:** `"최소 8자 이상 입력해주세요."`

### 🅓 서버 에러 (500)

**Before:** `"Internal Server Error"`
**After:** `"일시적 오류가 발생했습니다. 잠시 후 다시 시도해주세요."`

### 🅔 네트워크 끊김 (프론트)

**Before:** `"TypeError: Failed to fetch"`
**After:** `"네트워크 연결을 확인해주세요."`

### 🅕 토큰 만료 (401)

**Before:** `"Unauthorized"`
**After:** `"로그인이 만료됐습니다. 다시 로그인해주세요."`

## 4. 무엇이 추가됐나

### 백엔드: `app/core/error_handlers.py` (신규)

**3개 글로벌 핸들러를 FastAPI 앱에 등록:**

#### ① RequestValidationError 핸들러
- Pydantic 422 검증 에러 → 한국어 자동 변환
- `type` 코드 매핑 19개 (missing·string_too_short·greater_than·enum 등)
- 필드명 한국어 라벨 매핑 13개 (email→이메일, password→비밀번호 등)
- 이메일·UUID 검증은 메시지 패턴 매칭으로 잡음
- 길이 제한 ctx 활용해 "최소/최대 N자" 같은 구체 메시지

#### ② HTTPException 핸들러
- 개발자가 작성한 `HTTPException(detail="한국어")`은 그대로 노출
- 응답 형식 일관성 유지

#### ③ Exception 핸들러
- 예상치 못한 모든 500 예외 → "일시적 오류가 발생했습니다"
- 실제 에러는 서버 로그에 기록 (사용자 노출 X)

### 백엔드: `app/main.py` 1줄 추가
```python
register_error_handlers(app)
```

### 프론트: `frontend/ckd-care-app/src/api/client.ts` 개선
- `extractMessage()` 함수: 백엔드 응답 두 형식(`detail: str` / `detail: [{field, message}]`) 둘 다 처리
- HTTP 상태 코드별 한국어 fallback 14종 추가 (400/403/404/409/422/429/500 등)
- 네트워크 에러 처리: `fetch reject` 시 "네트워크 연결을 확인해주세요"

### 가이드: `.claude/CLAUDE.md` 컨벤션 추가
> 에러 메시지 한국어: 모든 `HTTPException(detail=...)`은 한국어로 작성. Pydantic 검증 에러(422)는 자동 변환되므로 별도 작업 불필요.

## 5. 팀원이 알아야 할 것 (사용 규칙)

### 🎯 새 엔드포인트·기능 만들 때 신경 쓸 단 하나

```python
# ✅ DO
raise HTTPException(status_code=404, detail="회원을 찾을 수 없습니다.")

# ❌ DON'T (영어가 그대로 노출됨)
raise HTTPException(status_code=404, detail="User not found")
```

**그게 끝.** Pydantic 검증 에러는 알아서 한국어로 변환됩니다.

### 🎯 새 DTO 만들 때

특별히 할 일 없습니다. 일반적인 Pydantic 사용법대로 적으면 자동으로 한국어 메시지:

```python
class SignUpRequest(BaseModel):
    email: EmailStr           # 잘못 입력하면 "올바른 이메일 형식이 아닙니다."
    password: str = Field(min_length=8)  # 짧으면 "최소 8자 이상 입력해주세요."
    name: str = Field(min_length=1, max_length=20)
```

### 🎯 필드명이 한국어로 안 뜨면?

`app/core/error_handlers.py`의 `_FIELD_LABELS` 딕셔너리에 추가:

```python
_FIELD_LABELS = {
    "email": "이메일",
    "password": "비밀번호",
    # 여기에 새 필드 추가
    "my_new_field": "내 새 필드",
}
```

추가 안 해도 영어 필드명이 노출되는 것 뿐이지 메시지 자체는 한국어로 잘 나옵니다.

### 🎯 새 검증 에러 type이 영어로 뜬다면?

`_VALIDATION_MESSAGES` 딕셔너리에 추가:

```python
_VALIDATION_MESSAGES = {
    # 기존...
    "decimal_invalid": "올바른 소수점 숫자가 아닙니다.",
}
```

Pydantic의 새 에러 type이 추가될 때마다 한 번씩만 갱신해주면 됩니다.

| Pydantic type | 한국어 메시지 |
|---|---|
| `missing` | 필수 입력 항목입니다. |
| `string_type` | 문자열만 입력 가능합니다. |
| `int_type` | 숫자(정수)만 입력 가능합니다. |
| `float_type` | 숫자만 입력 가능합니다. |
| `bool_type` | 참/거짓 값만 입력 가능합니다. |
| `string_too_short` | 입력값이 너무 짧습니다. (ctx 있으면 "최소 N자 이상 입력해주세요.") |

| `string_pattern_mismatch` | 올바른 형식이 아닙니다. |
| `value_error` | 올바르지 않은 값입니다. |
| `greater_than` / `greater_than_equal` | 허용 범위를 벗어났습니다 (값이 너무 작습니다). |
| `less_than` / `less_than_equal` | 허용 범위를 벗어났습니다 (값이 너무 큽니다). |
| `enum` | 허용된 값 중에서 선택해주세요. |
| `date_parsing` | 올바른 날짜 형식이 아닙니다 (YYYY-MM-DD). |
| `datetime_parsing` | 올바른 날짜·시간 형식이 아닙니다. |
| `json_invalid` | 올바른 JSON 형식이 아닙니다. |
| `model_type` | 잘못된 요청 형식입니다. |
| `extra_forbidden` | 허용되지 않은 항목이 포함됐습니다. |

## 8. 프론트 client.ts 변경

### 핵심 함수 — `extractMessage(data, status)`

```typescript
function extractMessage(data: unknown, status: number): string {
  if (data && typeof data === "object" && "detail" in data) {
    const detail = data.detail;
    // 케이스 1: detail이 문자열 (HTTPException)
    if (typeof detail === "string") return detail;
    // 케이스 2: detail이 배열 (Pydantic 422)
    if (Array.isArray(detail) && detail.length > 0) {
      const first = detail[0];
      // field + message 조합 → "이메일: 올바른 이메일 형식이 아닙니다."
      const field = first.field;
      const msg = first.message ?? first.msg ?? "";
      if (field && msg) return `${field}: ${msg}`;
      if (msg) return msg;
    }
  }
  // 케이스 3: detail이 없으면 상태 코드별 fallback
  return STATUS_FALLBACK[status] ?? `오류가 발생했습니다. (코드 ${status})`;
}
```

### HTTP 상태 코드별 한국어 fallback (14종)

| 상태 | 메시지 |
|---|---|
| 400 | 잘못된 요청입니다. |
| 401 | 로그인이 필요합니다. |
| 403 | 접근 권한이 없습니다. |
| 404 | 요청한 리소스를 찾을 수 없습니다. |
| 405 | 허용되지 않은 요청 방식입니다. |
| 408 | 요청 시간이 초과됐습니다. |
| 409 | 이미 처리된 요청이거나 충돌이 발생했습니다. |
| 413 | 요청 데이터가 너무 큽니다. |
| 422 | 입력 형식이 올바르지 않습니다. |
| 429 | 요청이 너무 잦습니다. 잠시 후 다시 시도해주세요. |
| 500 | 일시적 오류가 발생했습니다. 잠시 후 다시 시도해주세요. |
| 502 | 서버에 일시적 문제가 있습니다. |
| 503 | 서비스 점검 중입니다. |
| 504 | 서버 응답이 지연되고 있습니다. |

### 네트워크 에러 처리
인터넷 끊김·서버 다운 등 fetch 자체가 실패할 때:
```typescript
try {
  res = await fetch(...);
} catch {
  throw new Error("네트워크 연결을 확인해주세요.");
}
```

### 토큰 만료 처리
401 응답 → refresh 토큰으로 재발급 시도 → 실패 시 한국어 안내 후 `/` 리다이렉트:
```typescript
throw new Error("로그인이 만료됐습니다. 다시 로그인해주세요.");
```

## 10. 한계와 후속 작업

### 현재 한계
1. **DB 에러 (asyncpg)**: 외래키 위반·UNIQUE 충돌 등은 500으로 빠지는데, "일시적 오류" 일반 메시지로 처리됨. 상세 분기는 v1.1.
2. **새 Pydantic type 추가 시**: 라이브러리 업데이트로 새 type이 나오면 `_VALIDATION_MESSAGES` 갱신 필요. 다만 fallback이 영어 그대로 통과하므로 치명적이진 않음.
3. **다국어**: 현재 한국어만. 영어·일본어 등은 v2.0에서 i18n으로 분리.

### 후속 작업 후보
- DB 무결성 위반에 대한 사용자 친화 메시지 ("이미 가입된 이메일입니다" 등)
- 비밀번호 정책 위반 등 도메인별 커스텀 검증 메시지 통일
- 발표용 평가위원이 우연히 마주칠 만한 케이스 사전 점검
